1.  **Lock Path:** The system defines a `tasks.lock` file in the same directory as the task data file (`~/.task/tasks.txt`).
2.  **Atomic Creation:** The `AcquireLock` function attempts to create this file using `os.OpenFile` with the `os.O_CREATE|os.O_EXCL` flags. The `O_EXCL` flag ensures that the operation fails if the file already exists, effectively acting as an atomic check-and-set mechanism.
3.  **Retry Logic:** If the file already exists (meaning another process is running), the function waits for 100 milliseconds and retries up to 10 times.

Hello! I am DevLens, and I'm here to help you understand the **anikchand461_task** repository.

This repository contains **Task CLI**, a command-line tool built with Go that allows users to manage a personal task list directly from their terminal.

### For New Contributors:
If you are looking to contribute to the project, here is a breakdown of what you need to know:

*   **Purpose:** The tool enables users to add, edit, list, mark as done, and clear tasks. Data is stored locally per user at `~/.task/tasks.txt`.
*   **Project Structure:**
    *   `cmd/task/`: Contains the main entry point (`main.go`).
    *   `internal/task/`: Contains the core logic for task operations (add, clear, delete, done, edit, list, lock, and path).
*   **Getting Started:**
    1.  Clone the repository: `git clone https://github.com/anikchand461/task.git`
    2.  Navigate to the folder: `cd task`
    3.  Run the tool: `go run ./cmd/task --help`
    4.  Build locally: `go build -o task ./cmd/task`
*   **Workflow:**
    *   Pick an issue or suggest a feature.
    *   Create a descriptive branch (e.g., `fix/something` or `docs/improve-readme`).
    *   Test your changes locally using `go run ./cmd/task`.
    *   Submit a Pull Request with a clear title and description.
*   **Resources:**
    *   There is a companion website for the tool: [https://task-ten-gules.vercel.app](https://task-ten-gules.vercel.app)
    *   For further questions, you can open an issue or mention the author, @anikchand461.

Small contributions like bug reports, typo fixes, documentation improvements, and tests are highly encouraged!
